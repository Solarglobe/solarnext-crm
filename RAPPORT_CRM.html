Verdict

Le CRM est riche, mais pas encore cohérent au niveau SaaS attendu.

Les risques principaux sont le premier démarrage, le pipeline, la navigation des paramètres, les permissions visibles/invisibles, le modèle Clients, les endpoints settings concurrents et une couche visuelle encore trop hétérogène.

La priorité n’est pas de “faire joli” en premier. Il faut d’abord corriger ce qui casse la confiance : onboarding, pipeline réel, RBAC, settings et navigation.

Audit détaillé

Constats, preuves, impacts et corrections

CritiqueOnboarding / accès CRM

L’onboarding peut être contourné

Preuve
Login.tsx et MfaVerify.tsx redirigent vers /onboarding, mais ProtectedRoute.tsx vérifie seulement l’authentification. Le shell CRM est exposé dans main.tsx.
Impact utilisateur
Un nouveau client peut arriver dans un CRM non configuré, avec entreprise, mail, équipe et pipeline incohérents.
Correction
Créer un garde global qui bloque toutes les routes CRM tant que onboardingCompleted=false.
CritiqueOnboarding / pipeline

L’étape Pipeline Kanban est une fausse promesse

Preuve
OnboardingStepPipeline modifie une liste locale stockée dans settings_json.onboarding.pipeline. Le vrai CRM lit pipeline_stages.
Impact utilisateur
L’utilisateur croit personnaliser son Kanban, mais le backend ne change pas. C’est une rupture de confiance immédiate.
Correction
Retirer l’étape ou créer une vraie API transactionnelle pipeline_stages.
CritiqueBackend / pipeline seed

Les nouvelles organisations peuvent recevoir un pipeline obsolète

Preuve
Le trigger historique de seed pipeline peut diverger de la migration V2. La conversion client dépend de stageCode === "SIGNED".
Impact utilisateur
Un lead déplacé en “Signé” peut ne pas devenir client si le stage a code=NULL.
Correction
Réparer le seed V2 et ajouter une migration idempotente pour les stages existants sans code.
CritiqueOnboarding / validation

Les étapes obligatoires peuvent être sautées

Preuve
La sidebar d’Onboarding.tsx permet d’aller à n’importe quelle étape. finish() valide seulement l’étape active.
Impact utilisateur
Un compte peut terminer le setup sans entreprise, mail ou équipe correctement configurés.
Correction
Verrouiller les étapes futures et valider tout le parcours avant completed=true.
ÉlevéBackend / RBAC

PATCH onboarding n’est pas protégé comme un réglage organisation

Preuve
/api/organizations/onboarding utilise verifyJWT seulement, alors que /settings et /security exigent org.settings.manage.
Impact utilisateur
N’importe quel utilisateur authentifié de l’organisation peut modifier l’état onboarding.
Correction
Exiger org.settings.manage ou limiter à l’admin créateur initial.
ÉlevéOnboarding / mail

Le test mail est simulé

Preuve
testMailConnection() utilise setTimeout puis marque tested=true.
Impact utilisateur
L’interface annonce une connexion validée sans aucune preuve backend.
Correction
Brancher une API réelle de test mail ou renommer l’étape en configuration ultérieure.
ÉlevéOrganisation / données

Le profil entreprise onboarding n’alimente pas les données officielles

Preuve
SIRET, RGE, couleur et adresse restent dans settings_json.onboarding.profile.
Impact utilisateur
Les informations saisies peuvent ne pas apparaître sur devis, factures ou PDF.
Correction
Mapper ces champs vers les colonnes/settings canoniques.
ÉlevéSettings

Plusieurs APIs settings peuvent diverger

Preuve
/api/organization/settings merge arbitrairement settings_json, tandis que /api/organizations/settings valide economics/quote/finance.
Impact utilisateur
Risque de réglages corrompus ou de numérotation devis désynchronisée.
Correction
Déprécier/verrouiller l’ancien endpoint et migrer les callers.
ÉlevéNavigation

Sécurité et audit log sont cachés

Preuve
/settings/security et /admin/audit-log existent dans le router mais ne sont pas clairement exposés.
Impact utilisateur
MFA, sessions actives et journal d’audit sont introuvables sans connaître l’URL.
Correction
Créer un hub Paramètres permissionnel.
ÉlevéClients

Le modèle Clients est ambigu

Preuve
/clients affiche une page clients, mais /clients/:id redirige vers /leads/:id.
Impact utilisateur
L’utilisateur ne sait pas si un client est une vraie fiche ou un lead converti.
Correction
Choisir un modèle officiel et aligner routes, libellés, recherche et liens.
DesignSaaS UI

La couche visuelle est trop hétérogène

Preuve
CSS legacy, tokens, primitives, overrides et styles inline cohabitent. Les primitives UI sont contournées par modules.
Impact utilisateur
Le CRM paraît assemblé par couches, pas comme un produit SaaS homogène.
Correction
Stabiliser l’ordre CSS, migrer boutons/modales/états vers primitives communes.
MoyenMobile / feedback

Tables trop larges, alertes natives, états inégaux

Preuve
Plusieurs tables imposent >900px, et des window.alert/window.confirm restent dans des parcours importants.
Impact utilisateur
Mobile peu naturel, feedback abrupt, qualité perçue inférieure.
Correction
Créer vues responsive, toasts/modales, et composant PageState.

Roadmap

Plan de correction par phases

Phase 1 - Onboarding fiable

Corriger contournement, progression, pipeline fictif, mail simulé, équipe brouillon et mapping entreprise.

  • Garde global onboarding.
  • Validation globale des étapes.
  • Retrait ou vraie API pipeline.
  • PATCH onboarding protégé RBAC.

Phase 2 - Navigation et paramètres

Remettre toutes les pages au bon endroit.

  • Hub Paramètres.
  • Sidebar permission-aware.
  • Inventaire des routes cachées/legacy.

Phase 3 - Parcours métier

Clarifier Leads, Clients, Devis, Factures, Documents et Mail.

  • Modèle Clients officiel.
  • Chaîne devis-facture-documents.
  • Dashboard actionnable.

Phase 4 - Niveau visuel SaaS

Unifier design system, responsive, états et microcopy.

  • Primitives UI.
  • PageState.
  • Suppression alertes natives.
  • Mobile lists.

Phase 5 - Tests et gouvernance

Verrouiller les régressions.

  • Matrice routes/API.
  • E2E critiques.
  • Tests backend RBAC/settings/pipeline.
  • Release gate CRM.
