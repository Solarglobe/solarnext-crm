Approche V1 Manuel — Ce qu'on veut faire

Principe : dessiner comme la maison et le toit

L'utilisateur dessine le chien assis en deux étapes, directement sur la photo satellite :

Étape 1 — Contour (4 clics + fermeture) : l'empreinte du chien assis sur le pan de toit. Outil MODE_DORMER_CONTOUR. Crée un objet dans CALPINAGE_STATE.roofExtensions[].

Étape 2 — Faitage (2 clics) : la ligne de faitage du chien assis, à l'intérieur du contour. Outil MODE_DORMER_RIDGE. Stocke rx.ridge.a et rx.ridge.b sur l'objet créé à l'étape 1.

Le 3D est reconstruit automatiquement. Il n'y a PAS besoin de dessiner les arêtes manuellement — elles sont auto-calculées depuis le contour + le faitage.

Géométrie 3D exacte — Ce que le code produit

Règle utilisateur confirmée

"Tout contour est collé au TOIT donc à 0cm — arête côté contour à 0 puis le point qui rejoint arête et faitage est lui sur élevé."

⚠ Le contour N'EST PAS un rectangle — c'est le KITE visible sur la photo satellite

L'utilisateur dessine la vraie forme du chien assis telle qu'elle apparaît depuis le satellite. Un chien assis vu du dessus forme un kite / losange : avant large (façade visible depuis la rue), qui se rétrécit vers l'arrière jusqu'au faitage court. Ce n'est PAS un rectangle. Le contour 4 points suit exactement cette forme kite. Le faitage est un segment court dessiné à l'intérieur du kite.

VUE SATELLITE (du dessus) — CE QUE L'UTILISATEUR DESSINE SUR LA PHOTO : Rue / avant (downslope) ↓ frontLeft ──────────── frontRight ← façade (côté LARGE du kite, au bas du pan) ╲ ╱ arête╲ ╱arête ← calculées automatiquement (BUG-C à corriger) ╲ ╱ ridgeLeft ────── ridgeRight ← FAITAGE (2 points dessinés, court) ╱ ╲ arête╱ ╲arête ╱ ╲ rearLeft ──────────── rearRight ← arrière (côté étroit ou apex, vers faîtage principal) ↑ Faîtage principal / arrière (upslope) → L'utilisateur dessine le contour [frontLeft, frontRight, rearRight, rearLeft] en suivant la vraie forme kite visible depuis le satellite. Puis il dessine le faitage [ridgeLeft, ridgeRight] à l'intérieur. Les 4 arêtes sont calculées et tracées automatiquement. ───────────────────────────────────────────────────────────────── VUE 3D (élévation au-dessus du pan de toit) : ridgeLeft ──── ridgeRight ← ÉLEVÉS de ridgeHeightRelM le long de la normale au pan ↑ ↑ │ (4 arêtes) │ ← montent de hauteur 0 au niveau du contour │ │ jusqu'à ridgeHeightRelM au niveau du faitage FL(0cm) FR(0cm) RL(0cm) RR(0cm) ← contour COLLÉ au pan (z = z_toit) RÈGLE HAUTEURS : frontLeft, frontRight, rearLeft, rearRight → hauteur relative = 0 cm (collés à la surface du pan) ridgeLeft → élevé de ridgeHeightRelM le long de la normale au pan ridgeRight → même règle FACES (4 faces planes, wallHeightM = 0) : [frontLeft, frontRight, ridgeLeft] ← triangle avant (apex ou façade) [rearLeft, rearRight, ridgeRight] ← triangle arrière (jonction toit) [frontLeft, ridgeLeft, ridgeRight, rearLeft] ← quad pan gauche (PLANAIRE ✓) [frontRight, rearRight, ridgeRight, ridgeLeft] ← quad pan droit (PLANAIRE ✓)

Comment le code calcule les hauteurs

Dans projectRoofExtensionToSupportPlane.ts — projectSourcePoint() : 

// Projecter un point 2D (pixel image) sur la surface du pan de toit
const xy  = imagePxToWorldHorizontalM(point.x, point.y, mpp, northDeg);
const z   = zOnPlaneEquationAtFixedXY(patch.equation, xy.x, xy.y);  // ← z du pan à ce XY
const base = { x: xy.x, y: xy.y, z };                               // ← 0cm : SUR le pan

// Élever le point de heightRelM le long de la normale au pan
const top  = add3(base, scale3(normal, heightRelM));                 // ← élevé si heightRelM > 0

// Pour les points de CONTOUR : heightRelM = 0  → top = base (sur le pan)
// Pour les points de FAITAGE : heightRelM = ridgeHeightRelM (ex. 0.80)  → élevé

Le code lit ridgeHeightRelM depuis rx.ridgeHeightRelM (défaut 0.8 défini dans pushRoofExtensionFromContour). L'utilisateur peut le modifier via le panneau latéral de hauteur.

BUG-C (P1) — Konva : les 4 arêtes auto ne sont pas tracées

P1 VISUEL KonvaContoursLayer.tsx lignes 314–378 (section roofExtensions V1)

Symptôme : Après le dessin du contour + faitage, la vue 2D satellite montre un rectangle orange (contour) et une ligne orange épaisse (faitage), mais PAS les 4 arêtes qui relient les coins du contour aux extrémités du faitage. La forme "kite/losange" est invisible.

Cause : Le code Konva V1 (lignes 314–378) trace le contour et le faitage, et les arêtes manuelles si elles existent (rx.hips.left, rx.hips.right). Mais il ne calcule PAS automatiquement les connexions contour_coin → endpoint_faitage.

Logique attendue : Si on a le contour kite (4 coins) et le faitage (2 endpoints), on peut déduire les 4 arêtes automatiquement :

VUE SATELLITE — KITE SHAPE (ce que l'utilisateur a dessiné)

  frontLeft ──────────── frontRight    ← façade (large)
       ╲                      ╱
   arête╲                    ╱arête   ← à tracer automatiquement
         ╲                  ╱
      ridgeLeft ──────── ridgeRight    ← faitage (court, 2 points)
         ╱                  ╲
   arête╱                    ╲arête
       ╱                      ╲
  rearLeft ──────────── rearRight     ← arrière (étroit ou apex)

4 arêtes à tracer automatiquement :
  frontLeft  → ridgeLeft    (2 coins les + proches de ridgeLeft  → ridgeLeft)
  frontRight → ridgeRight   (2 coins les + proches de ridgeRight → ridgeRight)
  rearLeft   → ridgeLeft
  rearRight  → ridgeRight

  (ou l'inverse selon la géométrie réelle — l'algorithme distance trie les cas)

Mais : Pour un V1 dessiné manuellement, on ne sait pas quel corner du contour correspond à FL/FR/RL/RR. Il faut calculer quel endpoint du faitage est "le plus proche du bord avant du contour" et quel est "le plus proche du bord arrière".

Algorithme :

// Dans KonvaContoursLayer.tsx — section roofExtensions (après ridge lines)
// Si on a contour (4 pts) + ridge (a, b), calculer les 4 arêtes

const pts4 = rx.contour.points;  // [p0, p1, p2, p3]
const rA   = rx.ridge.a;
const rB   = rx.ridge.b;

// Trouver les 2 coins les plus proches de rA, et les 2 plus proches de rB
const distTo = (p, r) => Math.hypot(p.x - r.x, p.y - r.y);
const sorted = pts4.map((p, i) => ({
  p, i,
  dA: distTo(p, rA),
  dB: distTo(p, rB)
}));

// Les 2 coins avec (dA - dB) le plus négatif → proches de rA = "apex"
sorted.sort((a, b) => (a.dA - a.dB) - (b.dA - b.dB));
const nearA = [sorted[0].p, sorted[1].p];  // → rA
const nearB = [sorted[2].p, sorted[3].p];  // → rB

// Tracer : nearA[0]→rA, nearA[1]→rA, nearB[0]→rB, nearB[1]→rB

Prompt 3/4 — Konva kite arêtes
Tu travailles sur le projet SolarNext, module calpinage — chien assis (dormer) V1 manuel.

CONTEXTE GÉOMÉTRIQUE :
Un chien assis à pignon vu du dessus forme un "kite/losange" :
- Le contour est un quadrilatère (4 coins)
- Le faitage est un segment intérieur

- Les 2 coins les plus proches de ridge.a sont reliés à ridge.a
