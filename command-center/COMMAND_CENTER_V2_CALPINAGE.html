🎨 Partie 1 — Audit Rendering 3D
Analyse complète des matériaux, géométries, lumières, depth buffer, panneaux et scène Three.js
CRITIQUE<Canvas key={cameraViewMode}> — Reset WebGL complet
Chaque basculement perspective ↔ orthographique détruit et recrée l'intégralité du contexte WebGL : tous les shaders recompilés, toutes les textures re-uploadées en VRAM, tous les buffers réalloués. Pour un simple changement de matrice de projection, c'est une opération de 200-500ms. La solution correcte est un switch impératif via useThree().camera sans toucher au Canvas.
// PROBLÈME actuel <Canvas key={cameraViewMode}> ← RESET WEBGL COMPLET <ViewerSceneContent /> </Canvas> // SOLUTION cible <Canvas> ← clé fixe, jamais de remount <DynamicCamera mode={cameraViewMode} /> ← switch impératif <ViewerSceneContent /> </Canvas>
CRITIQUEViewerSceneContent key={createdAtIso} — Rebuild Scène Total
Tout changement de scene.metadata.createdAtIso unmount/remount l'intégralité du sous-arbre Three.js : toutes les géométries, tous les matériaux, tous les meshes recréés. Sur une scène avec 20 panneaux + obstacles, cela génère 50-100 allocations GPU par mutation. La solution est une invalidation partielle par layer (RoofLayer, PanelsLayer, ObstaclesLayer) avec des clés indépendantes.
MAJEURMatériaux — Rendu Trop Mat (Flat)
Les matériaux actuels : building shell (roughness 0.88, metalness 0.05), roof (roughness 0.64, metalness 0.16), obstacles chimney (roughness 0.88, metalness 0.03). Ces valeurs produisent un rendu quasi-diffus très flat, sans profondeur visuelle ni réflexions subtiles. Pour un rendu premium : shell 0.55/0.12, roof 0.45/0.22, chimney 0.65/0.08. Les panneaux PV (roughness 0.42, metalness 0.26) sont plus proches de la cible mais l'émissif est trop faible.
MAJEURShadow Map — Résolution Insuffisante
La shadow map est à 1024px en mode technique, 2048px en mode présentation. Pour un SaaS premium sur écrans Retina, 1024px produit des ombres pixelisées visibles. La valeur cible est 4096px avec shadow-camera-near=0.5 et une frustum optimisée à la bounding box de la scène. Le shadowBias=-0.00025 peut aussi causer du shadow acne sur les surfaces à faible pente.
MAJEURZ-Fighting — Système Polygonoffset Chaotique
16 couches de polygonOffset différentes, ajoutées au fil des patches, sans registre centralisé. Les valeurs vont de +1 (shell pousse en arrière) à -4 (éléments les plus en avant). Le renderOrder va de 0 à 32 avec des gaps (pas de valeurs 2-6, 14-17, etc.). Ce système est impossible à maintenir et produit des artefacts visuels imprévisibles. Solution : un DepthRegistry canonique avec des couches nommées.
// Valeurs actuelles dispersées : shell : polygonOffset +1/+1 renderOrder 0 roofPatch : polygonOffset +1/+1 renderOrder 0 obstacle : polygonOffset -2/-2 renderOrder 7-9 panel : polygonOffset -1/-1 renderOrder 0 (20 en drag) panelCellLine: polygonOffset 0/0 renderOrder 1 (22 en drag) liveDrag : polygonOffset -4/-4 renderOrder 24 vertexMarker : polygonOffset -1/-1 renderOrder 18 // ← aucune cohérence, aucun registre
MAJEURCell Lines — N Draw Calls par Panneau
L'InstancedMesh rend N panneaux en 1 draw call. Mais chaque panneau génère un <lineSegments> séparé pour ses lignes de cellules = N draw calls supplémentaires. Pour 20 panneaux : 21 draw calls au lieu de 2. Pour 50 panneaux : 51 draw calls. Solution : un seul InstancedLineSegments ou une LineSegments unique avec toutes les lignes de cellules concaténées, actualisé en même temps que l'InstancedMesh.
MOYENBuilding Shell — Couleur Trop Sombre
La coque bâtiment utilise VIEWER_SHELL_MESH_HEX = couleur sombre (dark navy) avec roughness 0.88. Combinée à un éclairage d'ambiance faible, elle donne l'impression d'une maison "noire et sale". Solution : passer à un gris ardoise clair (ex: #8fa0b0) avec une légère désaturation teintée, roughness 0.55, et s'assurer que l'AO des normal-maps amplifie la perception 3D.
MOYENNormals — computeVertexNormals() sur obstacles
Les volumes obstacles utilisent computeVertexNormals() après triangulation. Sur des formes à facettes nettes (cheminée rectangulaire), les normales interpolées donnent un aspect arrondi non voulu. Solution : computeVertexNormals() + computeFaceNormals() ou géométrie avec vertex dupliqués aux arêtes vives (flat shading natif via flatShading: true).

⚡ Partie 2 — Audit Panneaux PV
Diagnostic root-cause des 6 bugs principaux. Sources identifiées dans SolarScene3DViewer.tsx.
CRITIQUEBug 1 — Panneau Fantôme à l'Ancienne Position
Root cause : Après un commit de déplacement PV, l'InstancedMesh conserve l'ancienne position du panneau pendant 1-2 frames RAF (temps que la scène 3D se rebuilde). Workaround actuel : pvRebuildBlockPanelIds masque les panneaux déplacés (scale=0) jusqu'au prochain rebuild. Ce masquage est une rustine qui peut laisser des panneaux invisibles si le rebuild rate. Root cause réelle : Il n'y a pas de mécanisme d'invalidation sélective — le rebuild est tout-ou-rien.
// Workaround actuel dans SolarScene3DViewer.tsx : // "masque les panneaux dans l'InstancedMesh jusqu'au rebuild" const pvLayout3DEffectiveHiddenIds = useMemo(() => { const base = new Set(pvRebuildBlockPanelIds); // ← panels masqués via scale=0 dans PvPanelInstanced return base; }, [pvRebuildBlockPanelIds, ...]); // Solution cible : invalidation sélective // → commitPanelMove() met à jour directement les matrices // de l'InstancedMesh AVANT le rebuild scène complet
CRITIQUEBug 2 — Panneau Invisible après Première Pose
Root cause : Après placePanelAt3dHit(), la scène 3D met 1-2 frames RAF à se rebuilder. Pendant ce délai, le nouveau panneau n'existe ni dans l'InstancedMesh (pas encore rebuildé), ni dans l'overlay PV (focusBlock pas encore défini). Le panneau est invisible. Workaround actuel : appel immédiat de selectPvBlockFrom3d(r.blockId, null) pour forcer selected=true dans l'overlay avant le rebuild. Fragile car dépend de l'ordre d'exécution RAF.
CRITIQUEBug 3 — Clignotements selon Angle Caméra
Root cause : Double représentation pv3dDragSession (useState) + pv3dDragSessionRef (useRef). L'overlay PV est calculé à partir de pv3dDragSession state, mais ses deps excluent scene intentionnellement. Quand la scène se rebuilde (ex: après un save), panelInstanceColors peut se recalculer alors que pv3dOverlayPanelById est stale → certains panneaux reçoivent des couleurs erronées pendant 1-2 frames → clignotement visible surtout quand la caméra révèle ces incohérences de rendu.
CRITIQUEBug 4 — Drag Figé
Root cause : pv3dDragSession est synchronisé entre useState et useRef via un useEffect. Sous React Concurrent Mode, la ref peut diverger du state pendant le batch rendering. Si un event pointer arrive pendant cette fenêtre, le handler lit la ref (ancienne valeur) alors que le state a été mis à jour, et le drag se termine prématurément ou reste figé. Solution : un seul useRef pour tout l'état du drag (pattern imperativ), avec un setState minimal uniquement pour déclencher le re-render.
CRITIQUEBug 5 — Mauvais Panneau Sélectionné / Déplacé
Root cause : L'InstancedMesh expose instanceId au raycasting Three.js, mais le système d'inspection lit e.object.userData[INSPECT_USERDATA_KEY]. Hack actuel : mutation de e.object.userData au moment du click pour injecter le panel ID correct. Si plusieurs events arrivent rapidement (ex: double-click), la mutation peut être écrasée entre la lecture et le traitement → mauvais panneau sélectionné.
// HACK actuel (fragile) : e.object.userData[INSPECT_USERDATA_KEY] = { kind: "PV_PANEL", id: String(panel.id), // ← écrit au moment du click }; onInspectClick(e); // ← lit userData immédiatement après // SOLUTION cible : // Passer le panelId directement dans le CustomEvent // sans muter userData (event data immutable)
MAJEURBug 6 — Changements Visuels selon Angle Caméra
Root cause : Les panneaux PV sont des quads coplanaires avec la surface du toit. Avec polygonOffset=-1/-1 normal et -3/-3 en mode drag, les panneaux peuvent apparaître partiellement sous la surface à certains angles rasants, créant une apparence de "bords visibles seulement" ou de contour. Ce n'est pas un bug de shader mais un artefact de projection géométrique + Z-buffer à précision finie. Solution : élever légèrement les panneaux (+2cm) perpendiculairement à la normale du pan, plutôt que de tout faire via polygonOffset.
🏗 Partie 3 — Audit Architecture 2D / 3D

Problème #1 — Source de Vérité Distribuée

La vraie source de vérité est window.CALPINAGE_STATE, une variable globale mutée directement par l'IIFE de 25k lignes. Le store Zustand est une copie dérivée, rafraîchie par events. La CalpinageRuntime est une façade qui re-lit window à chaque appel.

Conséquences :

  • Impossible de tester unitairement les composants React (ils lisent window)
  • Impossible de reproduire un état précis sans reproduire toute la session
  • Race conditions entre les events phase2/phase3 et les useState React
  • SolarScene3DViewer lit directement window.CALPINAGE_STATE dans ses event handlers
🔴
Problème #2 — Monolithe 25 291 Lignes

calpinage.module.js est une IIFE qui contient : la gestion du canvas 2D, les outils de dessin, la gestion des pans, la gestion des obstacles, la pose PV, le système de sauvegarde, le chargement API, les animations, les handlers clavier/souris, la logique de validation, le rendu 2D complet.

Découpage nécessaire (extraits) :

  • canvas2DRenderer.ts — rendu du canvas
  • roofDrawingTools.ts — outils de dessin
  • obstacleManager.ts — cycle de vie obstacles
  • pvPlacementEngine2D.ts — placement 2D
  • calpinageStateMachine.ts — la state machine centrale
  • calpinagePersistence.ts — save/load

Problème #3 — Triple Canvas Overlays

Le système utilise 3 canvas superposés :

  1. Canvas 2D legacy (#calpinage-canvas-el) — rendu principal 2D
  2. Konva Stage (overlay React) — contours, pans, obstacles, panneaux
  3. R3F Canvas (Three.js) — viewer 3D complet

La synchronisation entre ces 3 layers est gérée via polling setInterval, ResizeObserver, et window events. Tout désalignement de taille crée des artefacts de hit-testing (click au mauvais endroit).

🖱 Partie 4 — Audit UX / Interactions
Comportements premium manquants, feedback visuel, fluidité, ergonomie CAD-light
CRITIQUECamera Jump sur Switch Mode
Quand l'utilisateur passe de 3D à 2D (plan), le Canvas R3F est détruit et recréé. La position de caméra est réinitialisée. L'utilisateur perd son point de vue, son zoom, son orientation. Un outil CAD premium garde la continuité de caméra entre les modes. Solution : switch de projection (perspective ↔ ortho) sans toucher au Canvas, en interpolant la position caméra via un tween GSAP/anime.
MAJEURVertex Edit Z — Feedback Non Fluide
Quand l'utilisateur drag un sommet de toiture verticalement, le système : 1) reçoit l'event pointer, 2) appelle applyRoofVertexHeightEdit(), 3) émet calpinage:official-structural-change, 4) déclenche un rebuild complet de SolarScene3D, 5) remonte le composant ViewerSceneContent. Résultat : latence perceptible (100-300ms) à chaque pixel de drag. Un CAD premium met à jour la géométrie du mesh directement en mémoire GPU pendant le drag, et ne commit qu'au mouseup.
MAJEURHit Testing via window Globals
Cinq fonctions de hit-test sont exposées sur window : __CALPINAGE_KONVA_PAN_HIT__, __CALPINAGE_KONVA_OBS_HIT__, __CALPINAGE_KONVA_SV_HIT__, __CALPINAGE_KONVA_PANEL_HIT__, __calpinage_hitTestPan__. Le legacy IIFE les appelle directement. Ce design est untestable, impossible à typer, fragile aux remounts React. Solution : un HitTestService typé, injectable via Context.

🔬 Partie 5 — Audit Technique Profond
Dette technique, couplages excessifs, modules obsolètes, race conditions, cycles de render
CRITIQUE17 useState + 15 useRef dans SolarScene3DViewer
Inventaire complet des hooks d'état dans le composant principal. 3 state machines distinctes (Z-drag, PV-drag, inspection) co-habitent dans le même composant sans isolation. Chaque changement d'état déclenche un re-render qui réévalue tous les useMemo de géométrie. Solution : 3 hooks custom useRoofVertexZDrag, usePvPanelDrag3D, useSceneInspection avec leur état interne isolé.
MAJEURFallback Pipeline Phase3Viewer.js (Gelé mais Présent)
Le fichier phase3Viewer.js est gelé et marqué LEGACY mais reste dans le codebase. Il utilise window.THREE, son propre système de rendu non-R3F, ses propres matériaux. Tant qu'il existe, il peut être accidentellement réactivé. Il doit être supprimé et remplacé définitivement par SolarScene3DViewer.
MAJEUR22 Fonctions Geometry Builder dans le Viewer
Les fonctions roundChimneyBodyGeometry, volumeRingAt, chimneyBrickLineGeometry, roofObstacleDetailGeometries, imagePolygonToRoofMeshGeometry, etc. (22 fonctions pures) sont définies dans SolarScene3DViewer.tsx. Elles n'ont aucune dépendance React. Elles devraient être dans obstacleGeometryBuilders.ts + roofGeometryBuilders.ts — testables isolément.
MAJEURsetTimeout(16ms) hors RAF Loop
Le refresh de l'overlay PV utilise window.setTimeout(..., 16) (équivalent 1 frame à 60fps) au lieu de requestAnimationFrame. Sous 30fps ou sur onglet caché, cela crée des mises à jour désalignées avec le rendu Three.js. De même, le commit Z-drag utilise un requestAnimationFrame natif hors de R3F — peut interférer avec le scheduling R3F.
MAJEURESLint Disable react-hooks/exhaustive-deps (Stale Closures)
Au moins 3 // eslint-disable-next-line react-hooks/exhaustive-deps dans SolarScene3DViewer, tous justifiés par "éviter un recalcul en cascade". Ces désactivations masquent des stale closures réelles : si les données changent sans que la dépendance ne soit dans le tableau, le memo peut retourner des données périmées sans que React ne le sache.
MOYENImport Statement au Milieu du Fichier (ligne 235)
Un import apparaît à la ligne 235 du SolarScene3DViewer, après plusieurs définitions de fonctions. C'est invalide selon import/first ESLint. Signe que le fichier a été étendu de façon incrémentale sans revue structurelle.
